今天的 builders 讨论高度集中在 agent 架构、权限边界和产品体验上。共识很清楚：模型能力在快速上升，但真正决定可用性的，越来越是工作流设计、跨端协作和用户信任机制。

Aaron Levie 认为 agent 架构需要频繁重写，企业软件会被 AI 重新铺开

他强调，AI 模型进展太快，过去为了弥补上下文窗口限制而搭建的系统，几个月后就可能失效。对部署 agent 的团队来说，最佳实践正在按季度变化，重点也从“写软件”转向“设计系统、编排平台、持续升级流程”。他的判断是，未来几乎每个行业都会需要这类工程能力。

今天只有一档播客值得关注，但信息密度很高，几乎把 Anthropic 对 agent 的产品哲学讲透了。核心不是更大的模型口号，而是 local computer、权限边界、skills、memory 和 UX 如何一起把能力变成可交付的工作流。

Felix Rieseberg 解释 Claude Cowork 的真实定位：本地计算、可控权限和低摩擦 UX

Felix 把 Claude Cowork 描述成一套围绕本地电脑运行的 agent 系统，而不是把一切搬进云端的抽象平台。它通过 VM、文件夹授权、连接器和受控网络访问，把安全边界做在执行层；skills 和 memory 则被实现成 markdown 和 text files，说明复杂能力并不一定依赖复杂基础设施。更关键的判断是，trust 不是靠一次性“证明自己”，而是从清理桌面、定时任务这类小任务开始，逐步让用户把更多工作交给它。整场对话还反复强调，AI 产品的瓶颈越来越少是模型本身，更多是 UX、任务拆解、组织工作方式和 human taste。

Blogs

Anthropic 今天的两篇工程博客都很直接：一篇在修正 agentic eval 的测量偏差，另一篇在把 Claude Code 的权限模型往更高自治推进。前者强调 benchmark 分数不能脱离资源配置单独看，后者则说明“少点点击”必须以可解释的安全层为代价。

Anthropic 说明 agentic coding eval 的分数会被基础设施配置明显干扰

这篇文章用 Terminal-Bench 2.0 和 SWE-bench 的实验说明，资源 headroom、CPU/RAM 限制和容器杀死策略本身就会改变分数。Terminal-Bench 上，从严格 1x 到 uncapped 的总提升达到 6 个百分点，其中相当一部分来自 infra error 下降；而 SWE-bench 上 5x RAM 也只能带来 1.54 个百分点的提升。结论很明确：agentic eval 不是纯模型测试，基准差距不到几个百分点时，必须同时看硬件、沙箱、时间限制和 enforcement 方法。

  • https://www.anthropic.com/engineering/infrastructure-noise

Claude Code auto mode 用分类器替代人工审批，目标是减少权限疲劳

Anthropic 把 auto mode 定位在手动审批和 dangerously-skip-permissions 之间：输入侧有 prompt-injection probe，输出侧有 transcript classifier，尽量拦住越权、泄露和误操作，同时让其余操作直接执行。文中给出的数据很保守也很关键，完整流水线在真实流量上的 FPR 降到 0.4%，但对真实 overeager actions 的 FNR 仍有 17%，因此它不是高风险基础设施的人工审核替代品，而是面向低摩擦自治的折中方案。
