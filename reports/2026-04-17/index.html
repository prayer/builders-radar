这一期几乎把 Notion 的 agent 方法论说透了：MCP、CLI、Custom Agents、meeting notes 都只是接口，真正重要的是系统记录、权限边界、eval 和自修复回路。

Latent Space：Notion 的 Custom Agents、MCP 和“软件工厂”

最核心的结论是，Notion 不是把 agent 当成一个独立产品层，而是把它嵌进系统记录、权限、评测和工作流里。Simon Last 和 Sarah Sachs 反复强调，CLI 适合轻量、权限明确的 agent，MCP 适合窄场景和严格授权，但真正决定上限的是 harness、session、sandbox 和自验证循环，而不是单个协议本身。Notion 的做法是把 custom agents、meeting notes、agentic find 和任务数据库串成同一套系统：让 agent 写入数据库、读取会议记录、生成 preread，并在失败时通过 eval 和 manager agent 回路修复。它们也明确表示，不会把 fine-tuning 当成主路径，因为工具和接口变化太快，外层系统与可替换的工具抽象更重要。

Blogs

两篇博客都在讲同一件事：长任务 agent 的关键不在模型单点能力，而在把 session、harness、sandbox、权限和恢复机制做成稳定抽象。

Anthropic Engineering：Scaling Managed Agents

这篇文章把 Managed Agents 的核心抽象拆成 session、harness 和 sandbox，并强调三者必须解耦，长任务才能在容器失败、网络中断或模型升级时继续恢复。作者用“brain from hands”解释为什么执行环境、会话日志和凭证都不应绑死在同一个容器里，并给出 p50 TTFT 下降约 60%、p95 下降超过 90% 的结果。它传递的重点不是某一版实现，而是把 agent 运行时定义成可替换的接口。

  • https://www.anthropic.com/engineering/managed-agents

Claude Blog：Claude Managed Agents 已进入 public beta

Claude Managed Agents 的核心卖点是把 sandboxed code execution、checkpointing、credential management、scoped permissions 和 tracing 这些生产化负担从开发者手里拿走。官方强调它支持长时间后台会话、多 agent 协作和更强的治理能力，内部结构化文件生成测试里，成功率最高提升了 10 个百分点。对构建者来说，这意味着竞争焦点正在从 prompt 本身转向运行时、治理和可观测性。

  • https://claude.com/blog/claude-managed-agents
